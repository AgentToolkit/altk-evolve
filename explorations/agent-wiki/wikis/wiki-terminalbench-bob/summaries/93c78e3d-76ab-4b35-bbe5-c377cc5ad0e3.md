# Diagnose and fix a system-wide Python installation where pip cannot install packages.

Bob worked inside a Docker container (terminal-bench broken-python task) where pip3 raised `ModuleNotFoundError: No module named 'pip'`. It confirmed Python 3.13.1 was present, reproduced the failure with both `pip3` and `python3 -m pip`, then tried `ensurepip --upgrade`, which falsely reported pip already satisfied. After that, Bob chased a red herring: it inspected the pip3 wrapper, found the shebang `#!/usr/local/bin/python3.13` and the symlink chain python3 -> python3.13, and rewrote the shebang with sed assuming a path mismatch — but the symlink was valid and the failure persisted.

The real diagnosis came when Bob listed site-packages and saw only `pip-24.3.1.dist-info` with no actual `pip/` module directory: the package files had been deleted, leaving orphaned metadata that made ensurepip think pip was installed. Bob then bootstrapped get-pip.py. curl and wget were both absent, so it fell back to `urllib.request.urlretrieve` to fetch the script and piped it to python3. This uninstalled the stale 24.3.1 metadata and installed pip 26.1.2. A final `pip3 --version` confirmed the fix.

## Key turns

- Reproduced the failure: pip3 and `python3 -m pip` both report no module named pip on Python 3.13.1.
- ensurepip --upgrade misleadingly reported pip already satisfied in site-packages.
- Pursued a wrong theory that the pip3 shebang (python3.13) was broken and edited it with sed — no effect since the symlink was valid.
- Listed site-packages and found only pip-24.3.1.dist-info with no pip/ module directory — the actual root cause.
- curl and wget were both missing, so used python3 urllib to download get-pip.py.
- Ran get-pip.py, which removed the stale metadata and installed pip 26.1.2; verified with pip3 --version.
